Task schema

This is a somewhat exhaustive set of notes on possible fields in a Task schema.

As many fields have imprecise and potentially overlapping definitions, the particular names used below are arbitrary to a degree. For example, it's easy to argue that a field called "release" could be called "milestone" instead; such arguments are avoided altogether in these notes.

Guidebook likely should not support all of these fields as a more complex schema is likely to work against the design goal of a minimal, lightweight user-experience (unless this can be done all under the hood in a way does not complicate the app usage).

Not yet accounted for
  • Backlog
  • Archived, Deleted
  • Deferred
  • Duplicate tasks
  • Feedback, Wishlist

Core schema​

value​

The text of the task.

Alternately could be called name or title depending on the context and interpretation.

children​

Subtasks. Potentially infinitely recursive.

order​

The natural order given to the tasks when presented in a list. This is often an implicit value based on the storage of the children.

uuid​

A universally unique ID for the task

Allows for permalinks and other content-agnostic references.

name​

A human-friendly unique ID as an addition or alternative to uuid. Making it human-friendly may make it more difficult to ensure uniqueness.

Another complication is if the task changes in such a way that the name is no longer apt. Should the task be allowed to have multiple names? If the task changed that much, should the old task be archived and a new one created?

Status​

status​

At it's core, has two values: todo or done via a simple checklist metaphor.

Could be expanded to have a work-in-progress value.

The set of values could easily expand to others such as:

  • "deferred"
  • "archived"
  • "won't fix"
  • "as designed"
  • "needs design"
  • "needs priorization"
  • "not fixed"
  • "not ready"
  • "unconfirmed"

A larger list increases risk of overlapping semantics which makes it more difficult for users to know which status is right, i.e. more values can make the problem of what status is correct "under-constrained." Inconsistent categorization then makes the categorization less useful as other users have lower confidence in searches, filters, and reports on a given status as it may miss items they would have expected to find.

status may also have a workflow, i.e. "todo" -> "work in progress" -> "done". The status transitions may be fixed and not all transitions should be allowed. These transitions might also be a place where gating or approvals must occur to ensure steps are not skipped.

ready​

A single boolean indicating that a task has been confirmed, vetted, or otherwise approved and is ready to be worked on.

This might be used for:

  • Tasks that are requests that have not yet been prioritized by a Product Manager
  • Tasks that require design-work (UX, visual, engineering, etc.) before they can be started
  • Tasks that represent bugs that should be verified and prioritized before work is done on them

Readiness overlaps with priority as implicitly an unready task should be of lowest effective priority and any priority recorded on the task only represents its priority when it is ready.

Readiness is also arguably a status that precedes "todo".

progress​

Percent done. Semantically overlaps with status.

Definition​

description​

A long-form description of the task to be done. This in itself theoretically could have a schema.

definition of done​

definition of done is distinct from success criteria in that it only represents if the agreed upon work was completed, not that the anticipated outcome or impact was realized by the task being done.

This might also be called "acceptance criteria."

Categorization​

tags​

A set of values.

Considerations:

  • Case-sensitivity
  • Valid characters (e.g. #, _, whitespace, etc.)

attributes​

A set of key-value pairs.

Considerations:

  • Are values always strings? Only simple types? Objects with unbounded nesting?
  • Are empty values allowed?
  • Are attributes allowed in addition to tags? Or are tags simply a special-case of attributes?

Roadmap related​

priority​

P0, P1, P2, P3, P4, P5

dependencies​

Other tasks that must be complete before this task can be completed.

deadline​

A "must finish by" date that is agnostic of the effort. A more specific field such as release would subsume such a field.

milestone​

release​

A grouping attribute that aligns the task to a specific deadline. That deadline is not necessarily the deadline of the individual task, only that the task's own deadline must before or on the deadline of the release.

project​

A grouping around a particular, on-going or otherwise long-lasting effort. This could also be a product field. This is distinct from release in that there is no a distinct deadline associated with this categorization.

Could be handled as a tags value unless the UI/UX has specific semantic knowledge of what a project is.

category​

A generic grouping. A task can only have a single category.

component​

The implementation area of the task. For example, "Rendering Subsystem" or "Storage Layer" might be components.

Consideration:

  • Should this be a an attribute rather than a core field?
  • Are multiple components allowed?

Estimation related​

effort​

An estimate of how long a task will take.

This can get nuanced for larger tasks as a single number may bake in assumptions about the team size that's working on it, the fraction of that team's time, the expertise of the people working on the task, etc. Is this a "perfect" estimate (imagine everyone is fully focussed and has the expertise to do the work)? Or are these "real" estimates (the calendar time it will take given all the estimators assumptions about the reality around the task)?

If the value bakes in implicit assumptions such as the above, it can require training from users to avoid errors.

confidence​

How much risk is there? Is this a straightforward task that has been done many times and is just a matter of time to complete? Or is this a novel task that might not be possible, might have a poor estimate, might not have the value anticipated?

reach​

If this task is done, how big will be impacted? Does it affect all users of a product? This is distinct from impact as it should only account for how many not how much.

impact​

What is the value of the task being done? This is distinct from reach as a task that's highly valuable only to a few people is still high impact. This is why reach x impact is a better metric for overall importance.

Risk assessment​

risk​

A concrete line item of something that could go wrong along with some form of high, medium, low metric. Usually a qualitative metric rather than numeric probability is fine as the intent usually more for planning and communication than quantitative analysis.

Should be paired with a mitigation task to reduce the risk. If it's not worth associating a risk with a mitigation, then arguably it's not worth calling out the risk.

mitigation​

A task or set of tasks that are being done to either:

  1. Front-load the risk in the broader project so it will be clear if it is actuated or not
  2. Reduce the cost/impact of the risk if it is actuated
