Concepts​

This tutorial aims to call out the conceptually independent (or at least separable) parts of a rendering engine so that the overall code structure is easier to understand. These are the "objects" we want to think about that exist regardless of how they are implemented.

Example

We know our engine will have some sort of materials we want to render objects with. Therefore, "Material" is called out here. Contrastingly, we do not call out "shader programs" here as this is more an aspect of "how" the concept of a Material is handled in WGPU.

Let's start with a very loose list of the conceptual objects in a rendering engine:

  • Scene
    • Camera
    • Lights
    • Instances
      • Models
        • Geometry
        • Material
        • Transforms
  • Renderer
  • User input
  • Frame loop
    • Actions
  • File formats
note

What is a core concept versus what is an implementation detail is a sliding scale that depends on what you want to expose to the user of the engine! If your engine provides fine-grained control of shader programs to the user as major feature of the engine, then perhaps shader programs are a core concept. The above is only one take on the topic.

User input​

Action Queue​

The "Action Queue", which follows the "Gang of Four" Command design pattern is introduced in order to:

Pros
  • Decouple input from actions
  • Ease implemenation of:
    • Custom key bindings
    • New controller methods
    • Automated / simulated user input
  • Process engine changes in a well-defined, deterministic order in the pipeline
Cons
  • Multiple places to update for every new action

Implementation​

To implement this we introduce the following distinct pieces of code:

  1. An enum for all the possible actions
  2. An ActionQueue for storing actions generated from user input or other sources
  3. A "key mapper" for translating user input into actions
  4. An Action processor for running actions in the frame loop

Enumerating all actions​

In a smaller engine, having a well-defined set of actions can help keep the codebase understandable as this makes the API of the engine very clear. In a more complex engine, it may be useful to introduce dynamic actions so plugins may add new action types that the core engine is not aware of. For this tutorial, we'll stick to a fixed set of actions, as exemplified below:

#[derive(Clone)]
pub enum Action {
    MoveForward(f32),
    MoveBackward(f32),
    RotateRight(f32),
    RotateLeft(f32),
    MoveUp(f32),
    MoveDown(f32),
    MoveLeft(f32),
    MoveRight(f32),
    // etc.
}

Queuing up the actions​

In the context of a rendering engine, a key constraint is the actions will be processed at a well-known time during the frame rendering. In the simplest case, we could use a simple Vec that enqueues the actions and processes them all in order at the start of the frame, leaving the Vec empty after processing.

pub struct ActionQueue {
    queue: Vec<Action>,
    frame_queue: HashMap<String, Action>,
    persist_queue: HashMap<String, Action>,
}

However, in practice, it is often useful to have two additional variations of actions:

1. Actions types that can be enqueued at most once per frame. For example, if a "jump" action is triggered by the keyboard, we don't necessarily want it to be possible to also create a second "jump" action via a controller or joystick. One solution to this is to make each such action idempotent, but this can be error-prone as it needs to be implemented for each such action. Instead, per the code below, we introduce the idea of a frame_queue that only allows one action of a given type per frame.

2. Actions that persist until explicitly removed. Another special case is a user input that triggers the start / end of an action that is to be run every frame. For example, an input to "move forward" that is continually moves the player character forward until an explicit input to stop would add an action and want it to be processed every frame until it is removed. This is the purpose of persist_queue.

impl Action {
    // Provide a unique id to each action type so that we can match
    // against types, not just full values.
    pub fn to_type(&self) -> &'static str {
        use Action::*;
        match self {
            MoveForward(_) => "MoveForward",
            MoveBackward(_) => "MoveBackward",
            RotateRight(_) => "RotateRight",
            RotateLeft(_) => "RotateLeft",
            MoveUp(_) => "MoveUp",
            MoveDown(_) => "MoveDown",
            MoveLeft(_) => "MoveLeft",
            MoveRight(_) => "MoveRight",
        }
    }
}

impl ActionQueue {
    pub fn new() -> ActionQueue {
        ActionQueue {
            queue: Vec::new(),
            frame_queue: HashMap::new(),
            persist_queue: HashMap::new(),
        }
    }

    pub fn enqueue(&mut self, action: Action) {
        self.queue.push(action);
    }

    // Ensures at most one of the action type is in the queue for a
    // given frame. Subsequent calls replace prior actions of the same type.
    pub fn enqueue_frame(&mut self, action: Action) {
        self.frame_queue
            .insert(action.to_type().to_string(), action);
    }

    pub fn toggle_persist(&mut self, action: Action, enable: bool) {
        let key = action.to_type();
        if enable {
            self.persist_queue.insert(key.to_string(), action);
        } else {
            self.persist_queue.remove(key);
        }
    }

    // Drains all one-time actions and frame actions, clones all
    // persistent actions.
    pub fn drain_actions_for_frame(&mut self) -> Vec<Action> {
        let mut actions = Vec::new();
        actions.extend(self.queue.drain(..));
        actions.extend(self.frame_queue.drain().map(|(_, v)| v));
        actions.extend(self.persist_queue.values().cloned());
        actions
    }
}

Mapping input to actions​

The next component is the mapper that takes various input and creates the Actions. This is fairly straightforward and is simply a mapping from whatever input system is being used to the creation of an Action object.

use winit::keyboard::KeyCode;

use crate::action_queue::ActionQueue;

pub fn map_to_action(q: &mut ActionQueue, keycode: &KeyCode, is_pressed: bool) -> bool {
    use crate::action_queue::Action::*;

    const STEP: f32 = 0.1;

    match keycode {
        KeyCode::KeyW | KeyCode::ArrowUp => {
            q.toggle_persist(MoveForward(STEP), is_pressed);
        }
        KeyCode::KeyQ | KeyCode::ArrowLeft => {
            q.toggle_persist(RotateLeft(STEP), is_pressed);
        }
        KeyCode::KeyS | KeyCode::ArrowDown => {
            q.toggle_persist(MoveBackward(STEP), is_pressed);
        }
        KeyCode::KeyE | KeyCode::ArrowRight => {
            q.toggle_persist(RotateRight(STEP), is_pressed);
        }
        KeyCode::KeyR => {
            q.toggle_persist(MoveUp(STEP), is_pressed);
        }
        KeyCode::KeyF => {
            q.toggle_persist(MoveDown(STEP), is_pressed);
        }
        KeyCode::KeyA => {
            q.toggle_persist(MoveLeft(STEP), is_pressed);
        }
        KeyCode::KeyD => {
            q.toggle_persist(MoveRight(STEP), is_pressed);
        }
        _ => return false,
    };
    true
}

Executing Actions​

Lastly, at a well-defined point in the frame loop update, we need to process the actions that have been enqueued thus far.

In a more complex engine, it would be likely be beneficial to allow action handlers to be "pluggable" both for code organization purposes (e.g. isolating all camera related action handlers to one file) as well as for the introduction of extensibility.

    fn update_actions(&mut self) {
        let actions = self.action_queue.drain_actions_for_frame();

        use cgmath::InnerSpace;
        let cam = &mut self.scene.camera;
        let forward = cam.target - cam.position;
        let forward_norm = forward.normalize();
        let forward_mag = forward.magnitude();
        let right = forward_norm.cross(cam.up);
        let camera_close_to_target = forward_mag < 2.0;

        use action_queue::Action::*;
        for action in actions {
            match action {
                MoveForward(step) => {
                    if camera_close_to_target {
                        continue;
                    }
                    cam.target += forward_norm * step;
                    cam.position += forward_norm * step;
                }
                MoveBackward(step) => {
                    cam.target -= forward_norm * step;
                    cam.position -= forward_norm * step;
                }
                RotateLeft(step) => {
                    cam.position = cam.target - (forward + right * step).normalize() * forward_mag;
                }
                RotateRight(step) => {
                    cam.position = cam.target - (forward - right * step).normalize() * forward_mag;
                }
                MoveUp(step) => {
                    cam.target += cam.up * step;
                    cam.position += cam.up * step;
                }
                MoveDown(step) => {
                    cam.target -= cam.up * step;
                    cam.position -= cam.up * step;
                }
                MoveLeft(step) => {
                    cam.target -= right * step;
                    cam.position -= right * step;
                }
                MoveRight(step) => {
                    cam.target += right * step;
                    cam.position += right * step;
                }
            }
        }
    }
